AI AMAZE · Data Source & Reporting Template

使用行為數據源、報表欄位定義與可視化模板

本頁聚焦第二階段：定義 AI AMAZE 後台應收集哪些數據源、這些數據源可產生哪些頻率 / 類型 / 比例 / 報表欄位，以及用可視化模板示範未來後台可以如何呈現。此版本不是整理現有資料成果，而是建立資料設計與畫面方向。

1. 數據源從對話、檢索、語音、裝置、管理行為、成本等使用行為取得記錄。
2. 報表定義定義每張報表的頻率、維度、欄位、公式與判讀方式。
3. 可視化模板用範例圖表展示：需求意圖、知識覆蓋、品質漏斗、入口健康與 ROI。

1. 需要的數據源：從使用行為取得哪些有意義記錄

以下包含目前已有、需要標準化、以及目前沒有但建議新增的資料。

Conversation / Message Event

部分現有 / 建議補強

事件例：conversation_started, message_sent, response_returned, conversation_closed

欄位例：project_id, character_id, session_id, user_id_hash, timestamp, channel, language, role, message_text, response_id

用途：建立互動量、回合數、問題類型、尖峰時段、使用者任務路徑。

RAG / Knowledge Retrieval Trace

建議標準化

事件例：kb_search_started, kb_hit, kb_miss, retrieved_chunk_used

欄位例：query, rewritten_query, knowledge_id, record_id, retrieved_title, score, rank, used_in_answer, no_result_reason

用途：知道哪些知識有用、哪些問題找不到答案、哪些 KB 需要補。

Answer Quality / Review Event

建議新增

事件例：answer_reviewed, user_feedback, cs_label_added

欄位例：answer_id, label, issue_type, reviewer_role, confidence, expected_answer, correction_note

用途：產生正確率、部分正確率、錯誤原因分布、優化前後改善率。

Channel / Device Behavior

部分現有 / 建議補強

事件例：device_online, device_heartbeat, widget_opened, kiosk_interaction_started

欄位例：device_id, channel_type, location, app_version, online_status, last_seen_at, touch_count, idle_time_sec

用途：區分沒人用、設備離線、現場入口異常、載具設計摩擦。

Voice / Multimodal Event

建議新增

事件例：stt_started, stt_success, tts_success, voice_interrupted

欄位例：language, stt_latency_ms, tts_latency_ms, transcript, confidence, failure_reason, audio_duration_sec

用途：觀察語音體驗、外語使用、辨識失敗、現場噪音或互動中斷。

User Intent / Task Classification

建議由 AI 分類產生

事件例：intent_classified, task_completed, task_abandoned

欄位例：intent, sub_intent, task_stage, urgency, location_related, commerce_related, service_related

用途：把熱門詞升級為熱門任務，支援 UX、營運與 ROI 判斷。

Admin / Content Maintenance Event

建議新增

事件例：kb_record_created, kb_record_updated, prompt_published, script_test_run

欄位例：admin_id_hash, project_id, object_type, object_id, action, diff_summary, publish_status

用途：連結內容維護行為與回答品質變化，知道改善是否真的有效。

Cost / Resource Usage

建議新增或估算

事件例：llm_usage_recorded, stt_usage_recorded, tts_usage_recorded

欄位例：model, input_tokens, output_tokens, audio_seconds, unit_cost, estimated_cost, cache_hit

用途：支援 ROI：互動價值、人工節省、LLM/語音成本、專案毛利觀察。

2. 這些數據源要產生什麼資訊：報表欄位與定義

每張報表都要能追溯到 event / field / formula，避免只有漂亮圖表。
報表計算頻率 / 維度欄位定義 / 公式建議視覺
使用行為總覽日 / 週 / 月；專案 × 入口互動數、session 數、有效問題數、平均回合、活躍裝置、尖峰時段有效問題數 = user message 扣除 greeting / idle / system trigger；平均回合 = user+assistant message pair。KPI + 趨勢線 + 入口堆疊圖
需求 / 意圖分析日 / 週；專案 × intent意圖、問題數、佔比、成長率、代表性問題、關聯 KB、入口來源同一 session 內相同 intent 可去重或保留，需標明算法；佔比 = intent 問題數 / 有效問題數。Treemap / Pareto bar / 代表問題表
知識覆蓋與缺口日 / 週；intent × KBKB hit rate、miss rate、低分檢索率、未覆蓋問題 Top N、建議補 KB 主題KB hit = retrieved score >= threshold 且 used_in_answer=true；miss = no result 或 fallback。Coverage matrix + gap queue
回答品質週 / 月；角色 × intent通過率、部分通過率、錯誤率、低信心率、人工標註數、錯誤類型通過率 = reviewed pass / reviewed total；低信心可用 similarity 或 LLM judge confidence。品質漏斗 + issue stacked bar
入口 / 載具表現日 / 週；channel × device入口互動量、裝置在線率、最後心跳、啟動率、平均停留、操作中斷率裝置在線率 = 有心跳裝置 / 已註冊裝置；中斷 = session 無完成/超時。入口分布 + device health table
語音與多語體驗週 / 月；language × channel語言分布、STT 成功率、TTS 成功率、語音重試率、外語回答品質STT 成功率 = successful transcript / voice input attempts；多語品質需抽樣或 AI judge。語言 heatmap + voice reliability cards
ROI / 服務價值估算月；專案 × intent可替代人工問題數、估算節省分鐘、估算人工成本、AI 成本、淨效益、改善後增益淨效益 = 節省人工成本 - LLM/STT/TTS 成本；需可調整假設參數。ROI waterfall + assumption panel
