AI AMAZE · Data Source & Reporting Template

使用行為數據源、報表欄位定義與可視化模板

本頁聚焦第二階段：定義 AI AMAZE 後台應收集哪些數據源、這些數據源可產生哪些頻率 / 類型 / 比例 / 報表欄位，以及用可視化模板示範未來後台可以如何呈現。此版本不是整理現有資料成果，而是建立資料設計與畫面方向。

1. 數據源從對話、檢索、語音、裝置、管理行為、成本等使用行為取得記錄。
2. 報表定義定義每張報表的頻率、維度、欄位、公式與判讀方式。
3. 可視化模板用範例圖表展示：需求意圖、知識覆蓋、品質漏斗、入口健康與 ROI。

1. 需要的數據源：事件表單

一列一個事件，方便逐項討論：現在有沒有、從哪裡來、缺哪些欄位。

閱讀方式：這裡先用「表單 / 試算表」方式定義事件。每一列不是畫面功能，而是一筆應該被記錄的資訊。後續可以在「目前是否有」與「可能來源」欄逐項標記 AI AMAZE 現況。

事件系統應記錄的使用行為或狀態。

知識未命中kb_miss 沒有找到足夠相關知識，或分數低於門檻。 retriever 無結果或低分時。 RAG service query, top_score, miss_reason, intent, language, fallback_policy trace log 需新增/整理 是否有 miss_reason、top_score 哪些問題是知識缺口？ 待確認來源
引用知識片段retrieved_chunk_used 回答實際採用的 KB 片段；不同於只找到結果。 回答生成後記錄引用來源。 Prompt builderanswer generator response_id, record_id, chunk_id, title, score, content_version trace log 需標準化 是否能追到 chunk 與版本 回答依據是否可追溯？ 待確認來源

報表計算頻率 / 維度欄位定義 / 公式建議視覺
使用行為總覽日 / 週 / 月；專案 × 入口互動數、session 數、有效問題數、平均回合、活躍裝置、尖峰時段有效問題數 = user message 扣除 greeting / idle / system trigger；平均回合 = user+assistant message pair。KPI + 趨勢線 + 入口堆疊圖
需求 / 意圖分析日 / 週；專案 × intent意圖、問題數、佔比、成長率、代表性問題、關聯 KB、入口來源同一 session 內相同 intent 可去重或保留，需標明算法；佔比 = intent 問題數 / 有效問題數。Treemap / Pareto bar / 代表問題表
知識覆蓋與缺口日 / 週；intent × KBKB hit rate、miss rate、低分檢索率、未覆蓋問題 Top N、建議補 KB 主題KB hit = retrieved score >= threshold 且 used_in_answer=true；miss = no result 或 fallback。Coverage matrix + gap queue
回答品質週 / 月；角色 × intent通過率、部分通過率、錯誤率、低信心率、人工標註數、錯誤類型通過率 = reviewed pass / reviewed total；低信心可用 similarity 或 LLM judge confidence。品質漏斗 + issue stacked bar
入口 / 載具表現日 / 週；channel × device入口互動量、裝置在線率、最後心跳、啟動率、平均停留、操作中斷率裝置在線率 = 有心跳裝置 / 已註冊裝置；中斷 = session 無完成/超時。入口分布 + device health table
語音與多語體驗週 / 月；language × channel語言分布、STT 成功率、TTS 成功率、語音重試率、外語回答品質STT 成功率 = successful transcript / voice input attempts；多語品質需抽樣或 AI judge。語言 heatmap + voice reliability cards
ROI / 服務價值估算月；專案 × intent可替代人工問題數、估算節省分鐘、估算人工成本、AI 成本、淨效益、改善後增益淨效益 = 節省人工成本 - LLM/STT/TTS 成本；需可調整假設參數。ROI waterfall + assumption panel
