使用行為數據源、報表欄位定義與可視化模板

本頁依據 2026-05-22-ai-amaze-observability 的使用觀測內容，定義 AI AMAZE 後台應收集哪些數據源、這些數據源可產生哪些頻率 / 類型 / 比例 / 報表欄位，以及用可視化模板示範未來後台可以如何呈現。

內容依據以 2026-05-22-ai-amaze-observability 為正本；既有 Excel 已排除於本版範圍。
1. 數據源從對話、檢索、語音、裝置、管理行為、成本等使用行為取得記錄。
2. 報表定義定義每張報表的頻率、維度、欄位、公式與判讀方式。
3. 可視化模板用範例圖表展示：需求意圖、知識覆蓋、品質漏斗、入口健康與 ROI。

1. 需要的數據源：事件表單

一列一個事件，方便逐項討論：現在有沒有、從哪裡來、缺哪些欄位。

閱讀方式：這裡用「表格 / 事件清單」方式定義事件。每一列不是畫面功能，而是一筆應該被記錄的資訊。後續可以在「目前是否有」與「可能來源」欄逐項標記 AI AMAZE 現況。

事件系統應記錄的使用行為或狀態。
來源可能產生資料的系統位置。
關鍵欄位報表計算與追溯需要的最小欄位。
討論狀態用來開會確認：已有、部分有、需新增、需標準化。
事件 / Event Code這是什麼何時產生可能從哪裡來關鍵欄位資料型態目前狀態假設要確認什麼可回答的問題討論欄

報表計算頻率 / 維度欄位定義 / 公式建議視覺

需求 / 意圖分析日 / 週；專案 × intent意圖、問題數、佔比、成長率、代表性問題、關聯 KB、入口來源同一 session 內相同 intent 可去重或保留，需標明算法；佔比 = intent 問題數 / 有效問題數。Treemap / Pareto bar / 代表問題表
知識覆蓋與缺口日 / 週；intent × KBKB hit rate、miss rate、低分檢索率、未覆蓋問題 Top N、建議補 KB 主題KB hit = retrieved score >= threshold 且 used_in_answer=true；miss = no result 或 fallback。Coverage matrix + gap queue
回答品質週 / 月；角色 × intent通過率、部分通過率、錯誤率、低信心率、人工標註數、錯誤類型通過率 = reviewed pass / reviewed total；低信心可用 similarity 或 LLM judge confidence。品質漏斗 + issue stacked bar
入口 / 載具表現日 / 週；channel × device入口互動量、裝置在線率、最後心跳、啟動率、平均停留、操作中斷率裝置在線率 = 有心跳裝置 / 已註冊裝置；中斷 = session 無完成/超時。入口分布 + device health table
零售點擊與對話關聯分鐘 / 小時 / 日；專案 × UI 元素 × session分鐘級點擊、click-to-message window、點擊後提問率、回答後點擊率、未關聯點擊點擊關聯 = 在設定時間窗內連到 conversation/message/response 的 click。源頭 DB 需保留分鐘級或更細時間戳記，因 GA 匯入無法支援這個粒度。Click timeline + conversation-link table
遊戲與機台延伸使用日 / 週；專案 × 裝置 × 遊戲模組遊戲開始、完成、獎勵、重試、掃描、分數/結果分布、後續提問遊戲完成率 = completed game events / started game events；後續提問率 = 有 game event 且後續有 user_message_sent 的 sessions / game sessions。機台功能採用矩陣 + game funnel

後台測試 Resource 使用日 / 週；environment × test type × project測試次數、成功/失敗、關聯 prompt/KB/response、actor role、正式流量排除旗標正式使用報表預設排除 backend_test_resource_recorded；只有在比較正式流量與測試流量時納入。Backend resource table + test-result status cards
ROI / 服務價值估算月；專案 × intent可替代人工問題數、估算節省分鐘、估算人工成本、AI 成本、淨效益、改善後增益淨效益 = 節省人工成本 - LLM/STT/TTS 成本；需可調整假設參數。ROI waterfall + assumption panel
